## 1. 目的与适用范围

### 1.1 目的
本手册定义生产环境**紧急回滚**的标准操作流程，确保在发版失败、线上故障或异常情况下，能够在**最短时间内**将系统恢复到稳定状态，最大限度减少业务影响。

### 1.2 适用范围
| 场景 | 触发条件 | 回滚目标 |
|------|----------|----------|
| **发版失败回滚** | 发版后监控指标异常、验证不通过 | 上一稳定版本 |
| **线上故障回滚** | 新版本引入严重 Bug（P0/P1） | 上一稳定版本 |
| **配置错误回滚** | 配置变更导致服务异常 | 上一稳定配置 |
| **数据库变更回滚** | 数据库变更导致数据异常 | 变更前状态 |

### 1.3 回滚原则
| 原则 | 说明 |
|------|------|
| **快速优先** | 先回滚恢复，再排查根因 |
| **数据安全** | 回滚不丢失用户数据，必要时先备份 |
| **影响最小** | 选择影响面最小的回滚方式 |
| **可验证** | 回滚后必须验证服务恢复正常 |
| **记录完整** | 回滚操作全程记录，用于复盘 |

### 2.2 决策标准

| 级别 | 判定标准 | 决策 | 时限 |
|------|----------|------|------|
| **P0 紧急回滚** | 核心交易链路不可用、大面积用户受影响、资金风险 | **立即回滚，无需审批** | 2 分钟内开始 |
| **P1 快速回滚** | 重要功能不可用、部分用户受影响、无 workaround | **5 分钟内决策回滚** | 5 分钟内开始 |
| **P2 评估回滚** | 一般功能异常、有 workaround、影响可控 | 尝试修复，15 分钟无效则回滚 | 15 分钟内决策 |
| **P3 观察修复** | 轻微异常、无用户感知 | 不触发本手册，常规修复 | - |

**回滚决策口诀**：
> "P0 不犹豫，P1 五分钟，P2 看修复，超时必回滚"

## 3. 回滚前准备（Pre-Rollback）

### 3.1 信息确认清单
回滚执行前，必须确认以下信息：

| 序号 | 确认项 | 确认内容 | 确认人 |
|------|--------|----------|--------|
| 1 | 回滚版本 | 上一稳定版本号/镜像 Tag | 运维工程师 |
| 2 | 回滚包状态 | 镜像/包存在于仓库，可快速拉取 | 运维工程师 |
| 3 | 配置状态 | 上一稳定版本配置已备份 | 运维工程师 |
| 4 | 数据库状态 | 如涉及 DB 变更，回滚脚本已准备 | DBA |
| 5 | 数据备份 | 当前数据已备份（防止回滚数据丢失） | DBA/运维 |
| 6 | 影响面 | 确认回滚影响的服务列表 | 发版负责人 |
| 7 | 通知对象 | 确认需要通知的业务方/客服 | 发版负责人 |

### 3.2 快速信息获取
```bash
# 获取当前运行版本（Kubernetes）
kubectl get deployment <service-name> -o yaml | grep image

# 获取历史版本（Helm）
helm history <release-name> --max 5

# 获取上一稳定版本镜像 Tag
kubectl get pods -l app=<service-name> -o jsonpath='{.items[*].spec.containers[*].image}'

# 查看发版记录（查询工单系统或发布平台）
# 版本号格式：v2.3.0（上一稳定） vs v2.3.1（当前故障）
```

## 5. 各场景回滚操作手册

### 5.1 容器化应用镜像回滚（最常用）

**适用场景**：Kubernetes/Docker 环境，应用发版失败

**回滚步骤**：

```bash
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 1: 确认当前版本和回滚目标版本
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 查看当前 Deployment 使用的镜像
kubectl get deployment order-service -o jsonpath='{.spec.template.spec.containers[0].image}'
# 输出：registry.example.com/order-service:v2.3.1（故障版本）

# 查看历史版本（假设上一稳定版本是 v2.3.0）
helm history order-service
# 或查看 Git Tag/发版记录

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 2: 摘流 - 从负载均衡移除（可选）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 方式A：直接调整 Service Selector（如果支持）
kubectl patch service order-service -p '{"spec":{"selector":{"version":"v2.3.0"}}}'

# 方式B：使用 Istio/Traefik 权重调整
# istioctl pc route ... （根据实际网关调整）

# 方式C：直接删除 Pod 等待重建（简单场景）
# 注意：此方式会有短暂流量中断

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 3: 执行镜像回滚
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 方式A：kubectl rollout undo（推荐，最快速）
kubectl rollout undo deployment/order-service
# 或回滚到指定版本
kubectl rollout undo deployment/order-service --to-revision=2

# 方式B：直接修改镜像（精确控制）
kubectl set image deployment/order-service   order-service=registry.example.com/order-service:v2.3.0

# 方式C：Helm 回滚（如果使用 Helm）
helm rollback order-service 1  # 回滚到 revision 1

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 4: 验证回滚状态
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 查看回滚进度
kubectl rollout status deployment/order-service

# 确认 Pod 运行正常
kubectl get pods -l app=order-service
# 状态应为 Running，Ready 1/1

# 确认镜像版本正确
kubectl get pods -l app=order-service -o jsonpath='{.items[0].spec.containers[0].image}'
# 应为 v2.3.0

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 5: 健康检查
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 检查服务启动日志
kubectl logs -l app=order-service --tail=50

# 调用健康检查接口
curl http://order-service:8080/health
# 应返回 200 OK

# 检查依赖连接
curl http://order-service:8080/ready
# 应返回 200 OK（数据库、Redis 等连接正常）

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 6: 引流恢复
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 将服务重新加入负载均衡（如果 Step 2 做了摘流）
# 或确认 Service 已自动指向新 Pod

# 发送测试流量验证
curl http://gateway.example.com/api/orders -H "X-Canary: test"
# 确认响应正常

# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# Step 7: 监控观察（至少 15 分钟）
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 观察关键指标
# - 错误率 < 0.1%
# - P99 延迟 < 基线
# - QPS 恢复正常
# - 无异常日志

# 使用 Grafana 查看 Dashboard
# 链接：https://grafana.example.com/d/order-service
```

**预期耗时**：2-5 分钟

### 5.2 配置回滚

**适用场景**：配置变更导致服务异常（如 Nacos/Apollo/Consul 配置）

```bash
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
# 配置回滚步骤
# ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# Step 1: 从配置中心获取上一版本配置
# Nacos 示例
# 登录 Nacos 控制台，找到配置，查看历史版本，回滚到上一版本

# Apollo 示例
curl -X POST "http://apollo-config.example.com/apps/{appId}/clusters/{clusterName}/namespaces/{namespaceName}/releases"   -H "Content-Type: application/json"   -d '{
    "releaseTitle": "rollback-to-stable",
    "releaseComment": "回滚到上一稳定版本",
    "releasedBy": "ops-team",
    "isEmergencyPublish": true
  }'

# Step 2: 触发服务刷新配置（如果支持热刷新）
kubectl exec -it <pod-name> -- curl -X POST localhost:8080/actuator/refresh

# 或重启服务（如果不支持热刷新）
kubectl rollout restart deployment/order-service

# Step 3: 验证配置生效
kubectl exec -it <pod-name> -- cat /app/config/application.yml
# 确认配置值已恢复
```

**预期耗时**：1-5 分钟

### 5.3 数据库变更回滚

**适用场景**：数据库 DDL/DML 变更导致数据异常

> ⚠️ **警告**：数据库回滚风险最高，必须 DBA 参与，严禁单人操作！

```sql
-- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
-- 数据库回滚步骤（必须由 DBA 执行）
-- ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

-- Step 1: 备份当前数据（回滚前必须执行！）
mysqldump -h prod-db.example.com -u backup -p order_db >   /backup/rollback/order_db_$(date +%Y%m%d_%H%M%S).sql

-- Step 2: 评估回滚方式
-- ┌─────────────────────────────────────────┐
-- │ 变更类型      │ 回滚方式                  │
-- │──────────────┼──────────────────────────│
-- │ 新增字段/表   │ 一般不回滚，删除即可       │
-- │ 修改字段类型  │ 需要数据转换脚本           │
-- │ 删除字段/表   │ 从备份恢复（高风险）       │
-- │ 数据迁移      │ 反向迁移脚本               │
-- │ 索引变更      │ 删除/重建索引             │
-- └─────────────────────────────────────────┘

-- Step 3: 执行回滚脚本（示例：回滚新增字段）
-- 假设发版时执行了：ALTER TABLE orders ADD COLUMN new_field VARCHAR(50);
-- 回滚脚本：
ALTER TABLE orders DROP COLUMN new_field;

-- Step 4: 验证数据一致性
SELECT COUNT(*) FROM orders;
-- 对比发版前数据量

-- Step 5: 验证业务功能
-- 由业务验证人执行核心功能测试
```

**预期耗时**：5-30 分钟（取决于数据量）

### 5.4 组合回滚（代码+配置+数据库）

**适用场景**：发版涉及多种变更类型

**执行顺序**（关键！）：
```
1. 先回滚数据库（如需要）
   ↓
2. 再回滚应用代码（镜像回滚）
   ↓
3. 最后回滚配置
   ↓
4. 统一验证
```

> **原则**：数据库回滚必须在应用回滚之前，避免应用写入新格式数据导致不一致。

### 5.5 灾备切换（极端场景）

**适用场景**：回滚失败、基础设施故障、数据中心不可用

**触发条件**：
- 主回滚方式失败
- 基础设施（K8s 集群、网络）不可用
- 数据损坏无法修复

**执行步骤**：
```
1. 决策：值班主管 + 技术负责人共同决策启动灾备
2. 通知：通知所有相关方，启动重大故障应急响应
3. 切换：DNS/负载均衡切换至灾备环境
4. 验证：确认灾备环境服务正常
5. 恢复：主环境修复后，评估是否切回
```

> 详见《灾备切换手册》OPS-DR-001

| 序号 | 验证项 | 验证方法 | 通过标准 | 验证人 |
|------|--------|----------|----------|--------|
| 1 | 服务状态 | `kubectl get pods` | 全部 Running | 运维工程师 |
| 2 | 版本确认 | `kubectl describe pod` | 镜像 Tag 正确 | 运维工程师 |
| 3 | 健康检查 | `curl /health` | 返回 200 | 运维工程师 |
| 4 | 依赖连接 | `curl /ready` 或日志 | 无连接错误 | 运维工程师 |
| 5 | 错误率 | Grafana Dashboard | < 0.1% | 运维工程师 |
| 6 | 延迟指标 | Grafana Dashboard | P99 < 基线 | 运维工程师 |
| 7 | 业务功能 | 核心功能测试 | 全部通过 | 业务验证人 |
| 8 | 数据一致性 | 数据库校验 | 无异常 | DBA |

### 6.2 观察期要求
- **P0 服务回滚**：观察 30 分钟，确认完全稳定
- **P1 服务回滚**：观察 15 分钟
- **P2 服务回滚**：观察 10 分钟

## 8. 回滚工具与脚本

### 8.1 快速回滚脚本（Kubernetes）

```bash
#!/bin/bash
# rollback.sh - 快速回滚脚本
# 用法：./rollback.sh <service-name> <target-version>

SERVICE_NAME=$1
TARGET_VERSION=$2
NAMESPACE=${3:-default}

if [ -z "$SERVICE_NAME" ] || [ -z "$TARGET_VERSION" ]; then
    echo "用法: $0 <service-name> <target-version> [namespace]"
    exit 1
fi

echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "开始回滚: $SERVICE_NAME → $TARGET_VERSION"
echo "时间: $(date '+%Y-%m-%d %H:%M:%S')"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"

# 1. 获取当前版本
echo "[1/5] 获取当前版本..."
CURRENT_IMAGE=$(kubectl get deployment $SERVICE_NAME -n $NAMESPACE -o jsonpath='{.spec.template.spec.containers[0].image}')
echo "当前版本: $CURRENT_IMAGE"

# 2. 执行回滚
echo "[2/5] 执行镜像回滚..."
kubectl set image deployment/$SERVICE_NAME   $SERVICE_NAME=registry.example.com/$SERVICE_NAME:$TARGET_VERSION   -n $NAMESPACE

# 3. 等待回滚完成
echo "[3/5] 等待回滚完成..."
kubectl rollout status deployment/$SERVICE_NAME -n $NAMESPACE --timeout=300s

# 4. 验证版本
echo "[4/5] 验证版本..."
NEW_IMAGE=$(kubectl get pods -l app=$SERVICE_NAME -n $NAMESPACE -o jsonpath='{.items[0].spec.containers[0].image}')
echo "新版本: $NEW_IMAGE"

if [[ "$NEW_IMAGE" == *"$TARGET_VERSION"* ]]; then
    echo "✅ 版本验证通过"
else
    echo "❌ 版本验证失败！"
    exit 1
fi

# 5. 健康检查
echo "[5/5] 健康检查..."
POD_NAME=$(kubectl get pods -l app=$SERVICE_NAME -n $NAMESPACE -o jsonpath='{.items[0].metadata.name}')
HEALTH_STATUS=$(kubectl exec -n $NAMESPACE $POD_NAME -- curl -s -o /dev/null -w "%{http_code}" http://localhost:8080/health)

if [ "$HEALTH_STATUS" == "200" ]; then
    echo "✅ 健康检查通过 (HTTP 200)"
else
    echo "⚠️ 健康检查异常 (HTTP $HEALTH_STATUS)"
fi

echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
echo "回滚完成: $(date '+%Y-%m-%d %H:%M:%S')"
echo "━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━"
```

### 8.2 一键回滚命令速查

| 场景 | 命令 | 预期耗时 |
|------|------|----------|
| K8s Deployment 回滚 | `kubectl rollout undo deployment/<name>` | 2-3 min |
| Helm 回滚 | `helm rollback <release> <revision>` | 2-3 min |
| 指定版本回滚 | `kubectl set image deployment/<name> <container>=<image:tag>` | 2-3 min |
| 配置回滚（Nacos） | 控制台操作或 API 调用 | 1-2 min |
| 数据库回滚 | 执行回滚 SQL 脚本 | 5-30 min |
