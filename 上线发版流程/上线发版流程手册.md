# 上线发版流程手册（标准模板）

> **文档编号**：OPS-REL-001  
> **版本**：v1.0  
> **生效日期**：2026-06-08  
> **适用范围**：生产环境（Production）发版  
> **维护部门**：运维部 / SRE 团队

---

## 1. 目的与适用范围

### 1.1 目的
本手册旨在规范生产环境的上线发版流程，确保发版过程**可控、可追溯、可回滚**，最大限度降低发版风险，保障系统稳定性。

### 1.2 适用范围
- 所有涉及生产环境代码/配置变更的发布
- 包括：应用发版、配置变更、数据库变更、基础设施变更
- 不包括：日常监控告警处理、故障应急修复（参见《应急响应手册》）

### 1.3 基本原则
| 原则 | 说明 |
|------|------|
| **灰度发布** | 先小流量验证，再全量发布 |
| **可观测** | 发版全程监控，关键指标实时可见 |
| **可回滚** | 任何发版必须在 5 分钟内可回滚 |
| **双人复核** | 关键操作必须双人确认 |
| **变更窗口** | 仅限低峰期（通常 02:00-06:00）执行 |

## 2. 角色与职责

| 角色                   | 职责             | 人员         |
| -------------------- | -------------- | ---------- |
| **发版负责人（PM）**        | 整体协调、风险评估、决策拍板 | 项目经理/技术负责人 |
| **发版执行人（Executor）**  | 具体执行发版操作       | 运维工程师/SRE  |
| **复核人（Reviewer）**    | 关键步骤复核、确认      | 高级运维/架构师   |

| **值班负责人（Duty）**      | 应急决策、重大故障升级    | 值班主管       |

## 3. 发版前准备（Pre-Release）

### 3.1 发版申请与审批
```
[发起] 发版负责人在工单系统提交《生产发版申请单》
    ↓
[评审] 技术评审会（涉及架构变更时）
    ↓
[审批] 部门负责人审批（P0服务需总监级审批）
    ↓
[排期] 确定发版窗口时间，通知相关方
```

**发版申请单必填内容**：
- [ ] 发版范围（服务清单、影响面）
- [ ] 变更内容（代码版本、配置变更、SQL 脚本）
- [ ] 回滚方案（详细回滚步骤、预计耗时）
- [ ] 验证方案（验证项、验证方法、通过标准）
- [ ] 监控方案（关键指标、告警阈值、观察时长）
- [ ] 应急预案（异常场景、处理措施、联系人）

### 3.2 环境检查清单

**检查项（Checklist）**：

| 序号 | 检查项 | 标准 | 检查人 | 结果 |
|------|--------|------|--------|------|
| 1 | 测试环境验证 | 测试通过，无 P0/P1 级 Bug | 测试工程师 | ☐ |
| 2 | 代码 Review | 通过 Code Review，合并到 Release 分支 | 技术负责人 | ☐ |
| 3 | CI/CD 流水线 | 构建成功，镜像已推送仓库 | 运维工程师 | ☐ |
| 4 | 配置一致性 | 生产配置与预发环境一致且已审计 | 运维工程师 | ☐ |
| 5 | 数据库脚本 | DBA 审核通过，已准备回滚脚本 | DBA | ☐ |
| 6 | 依赖服务状态 | 下游服务无发版计划，状态正常 | 运维工程师 | ☐ |
| 7 | 监控告警状态 | 当前无未恢复告警 | 值班人员 | ☐ |
| 8 | 备份状态 | 数据库/配置已备份（24小时内） | 运维工程师 | ☐ |
| 9 | 回滚包就绪 | 上一版本镜像/包已标记，可快速回滚 | 运维工程师 | ☐ |
| 10 | 值班人员确认 | 发版时段值班人员已确认在岗 | 值班主管 | ☐ |

### 3.3 通知与公告
- **提前 24 小时**：邮件通知相关团队（业务方、客服、运维、DBA）
- **提前 2 小时**：在企业微信群/钉钉群再次确认
- **发版前 30 分钟**：确认所有参与人员在线，召开简短站会（5分钟）

### 4.2 详细执行步骤

#### Step 1: 摘流（Traffic Drain）
- **操作**：从负载均衡器（Nginx/SLB/K8s Ingress）移除目标实例
- **确认**：`curl` 验证实例不再接收新流量
- **等待**：等待存量请求处理完成（通常 30-60 秒）
- **记录**：记录操作时间、操作人

#### Step 2: 部署（Deploy）
- **操作**：
  - 容器化环境：`kubectl set image` 或 Helm upgrade
  - 虚拟机环境：Ansible 发布脚本 / 手动替换包
- **确认**：部署命令返回成功，Pod/服务状态正常
- **记录**：记录版本号、镜像 Tag、部署耗时

#### Step 3: 健康检查（Health Check）
- **检查项**：
  - [ ] 服务启动成功，无 Error 日志
  - [ ] 健康检查接口返回 200（如 `/health`）
  - [ ] 依赖服务连接正常（数据库、Redis、下游服务）
  - [ ] 关键指标无异常（CPU、内存、GC）
- **超时**：5 分钟未通过则判定失败，触发回滚

#### Step 4: 引流（Traffic Enable）
- **操作**：将实例重新加入负载均衡
- **策略**：
  - 灰度：先导入 5% 流量，观察 10 分钟
  - 全量：逐步增加至 100%，每步 25%，间隔 5 分钟
- **确认**：监控流量分布正常，无 5xx 错误激增

#### Step 5: 监控（Monitor）
- **监控时长**：P0 服务至少观察 30 分钟，P1 服务 15 分钟
- **关键指标**：
  | 指标 | 告警阈值 | 观察工具 |
  |------|----------|----------|
  | 错误率 | > 0.1% | Grafana + Alertmanager |
  | P99 延迟 | > 基线 150% | Grafana |
  | QPS | 波动 < 20% | Grafana |
  | CPU/内存 | < 80% | Prometheus |
  | 业务指标 | 无异常下跌 | 业务大盘 |

#### Step 6: 验证（Validate）
- **业务验证**：执行验证用例（核心功能 Smoke Test）
- **技术验证**：接口测试、数据一致性检查
- **签字确认**：业务验证人在工单系统确认"验证通过"

#### Step 7: 观察（Observe）
- **观察期**：发版后 2 小时内保持高度警惕
- **值班交接**：发版信息完整交接给下一班值班人员
- **回滚窗口**：发版后 4 小时内保留回滚能力（不清理旧版本资源）

## 5. 不同服务级别的发版策略

### 5.1 P0 级服务（核心交易链路）
```
发版策略：金丝雀发布（Canary）

流量比例：
  0% → 5%（10分钟）→ 25%（10分钟）→ 50%（10分钟）→ 100%

要求：
  - 必须双人复核每个流量切换步骤
  - 每个阶段必须人工确认指标正常
  - 任何异常立即回滚，不尝试修复
```

### 5.2 P1 级服务（重要支撑服务）
```
发版策略：滚动发布（Rolling Update）

批次策略：
  第一批：1 个实例（验证）
  第二批：50% 实例
  第三批：100% 实例

要求：
  - 批次间至少间隔 5 分钟
  - 自动监控，异常自动暂停
```

### 5.3 P2 级服务（一般服务）
```
发版策略：全量发布（Blue-Green）

流程：
  1. 部署新版本（Green）
  2. 健康检查通过
  3. 切换流量到 Green
  4. 保留 Blue 环境 1 小时（快速回滚）

要求：
  - 可自动化执行
  - 保留回滚能力 1 小时
```

## 6. 发版后工作（Post-Release）

### 6.1 发版报告
发版完成后 24 小时内，发版负责人提交《发版总结报告》：
- 发版内容概述
- 发版时间线（实际 vs 计划）
- 遇到的问题及处理
- 监控指标对比（发版前后）
- 待优化项

### 6.2 归档
- 工单关闭，关联变更单号
- 发版脚本/配置归档到 Git
- 监控 Dashboard 截图存档

### 6.3 复盘（仅异常发版）
- 触发条件：发版过程中出现异常或回滚
- 复盘会：发版后 48 小时内召开
- 输出：改进措施，更新本手册
